## Session 15 — Teacher Notes

*Phase 3, Python basics · Session 15 of 16 · Title: Milestone
project work day 1*

### Purpose of this session

Same shape as Phase 1 Session 8. Plan + start building. Four
jobs, in priority order:

1. **Force every student to plan before coding.** Phase 3
   projects are bigger than Phase 1's; the planning step
   matters more.
2. **Help every student scope realistically.** With Python's
   full toolkit, kids will WANT to build ambitious things.
   Ambitious is fine; *unfinishable* is not. Push hard on
   "simplest version that's still cool."
3. **Set up Session 16's success.** Every kid leaves today
   with at least *something* running. Even a single working
   function counts.
4. **Make sure every kid has at least one idea.** Some come
   prepared; some don't. The seed list serves the latter
   group.

### Before class

**Bring:**

- Printed copies of the five planning questions for any kid
  without paper.
- A pen for sign-offs.
- Optional: printed seed-ideas list.

**Set up:**

- Demo machine ready to look at any earlier program for
  reference.
- Class roster with each student's name and (if you noted)
  any project ideas they mentioned in earlier sessions.

**Prep time:** ~15 minutes including printing. Plus: review
your Session 14 notes about which kids mentioned which
ideas during the text adventure session.

### Timing and flow

Total: ~90 min.

- **Welcome and intro** (~5 min) — what today is, what next
  week is, why planning matters.
- **Part A: planning** (~30 min) — silent planning ~12 min,
  Mr. Eric sign-off ~8 min (overlap with planning), buffer ~10
  min.
- **Build time** (~45 min) — pure build; you circulate.
- **Wrap-up** (~10 min) — "one thing I got working" round-robin.

If running short, **the wrap-up sharing can be cut to 5 min.**

### Teaching the planning step

The frame matters. Open with something like:

> "Today's the planning session. You're about to spend the
> next 90 minutes — plus most of next week's class — building
> something. The kids who plan first will get *way* further
> than the kids who just start typing. Five minutes of
> planning saves an hour of confusion. We're going to plan
> first."

Then have them work through the five questions silently for
~12 minutes. **Walk the room.** Read over shoulders.

Specific moves:

- **Stuck on Q1 (no idea):** Sit with them. Read the seed list
  together. Pick one. Modify slightly. Move on.
- **Wrote a paragraph for each (overscoping):** "Okay, this is
  ambitious. Question 4 — what's the simplest version? What
  would you build *first*?" Push hard.
- **Finished in 5 minutes (probably underscoping or
  over-confident):** Read theirs. If buildable, sign off. If
  missing something obvious (like persistence for a journal),
  ask about it.
- **Plan with no implementation path:** Walk through how each
  piece would work. If a piece doesn't have a Phase-3
  answer, simplify.

The Mr. Eric sign-off is real. Look for:
- Are all 5 questions answered?
- Is question 4 (simplest version) buildable in ~90 min?
- Do the data structures and functions in question 3 actually
  map to things they know how to build?

If yes, sign off. If no, ask one focused question.

### Teaching build time

Same as Phase 1 Session 8: walk the room, help individuals,
**don't take their keyboard.**

Triage during build:
- **Working code, polishing:** Let them.
- **Mostly working, has a bug:** Help them debug. Use the
  Thonny debugger when appropriate.
- **Has pieces but doesn't run:** Help get to runnable. Cut
  features ruthlessly.
- **Hasn't built much:** Most attention here. Help them build
  the simplest version.

Specific to Phase 3 (vs Phase 1):
- **Use the debugger.** Phase 3 students know it now (Session
  5). When a kid is stuck, suggest stepping through with the
  debugger.
- **Use the Reading Error Messages habit.** Errors will
  happen. Read them slowly together when they do.
- **Functions and modules.** If a kid's code is getting
  complicated, suggest pulling pieces into functions.

### Common stumbles

- **No idea for a project.** Most common; use the seed list.
- **Way too ambitious idea ("I want to build Roblox").**
  Common, especially among older kids. Push to question 4.
- **Plan with no clear "win" or end state.** "The user types
  things and the computer responds" — what's the win? What's
  the goal? If not a game, what's the user's goal?
- **Plan that requires concepts not yet covered.** Some kids
  will want to use graphics, sound, web, etc. Redirect: "all
  of those are coming in later phases. For now, what can you
  do with text-based Python?"
- **Frozen at the keyboard.** Same as Phase 1 — sit with them,
  help them write the first line, hand back the keyboard.

### Differentiation

- **Younger kids (9-10):** Push them toward the simpler seed
  ideas (mad libs, name generator). Help them scope down hard.
- **Older kids (12+):** Will overscope. Push question 4
  hard. The kid who wants "an MMO" needs to leave today
  having built "a single user typing a name and getting a
  response."
- **Advanced (any age):** Trust them with bigger scope. Ask
  "what's the riskiest part?" and have them build that
  *first*.
- **Struggling:** A kid who's been struggling all phase
  needs the scope-down conversation more than anyone. A
  trivia game with 3 questions is a totally valid milestone
  for them. Don't compare projects.

### What to watch for

- **Kids who wrote a great plan but can't start building.**
  Common. Help them write the first line.
- **Kids who skipped the plan.** Ask: "Show me your plan."
  If they don't have one, have them sit and write one.
  Don't let anyone code without a plan.
- **Kids who change their mind mid-session.** That's okay
  if it's earlier — re-plan. Not okay later — they need
  the rest of the time to build *something.*
- **Buddies trading ideas during planning.** Encourage —
  this is the talk-it-through habit.
